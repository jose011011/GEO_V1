CAMBIOS REALIZADOS (resumen)

Motivo del bug:
- Error al registrar profesional: SQLSTATE[23000]: Column 'tipo_documento' cannot be null.
- Se detectó desalineación entre el esquema/columnas esperadas en BD y el flujo de inserción para documentos.

====================================================
1) database/marketplace_servicios_lp.sql
====================================================
Se MODIFICÓ (sobrescrito completo).

QUITADO / reemplazado:
- Estructura anterior que usaba campos tipo_documento / ci_nit en la tabla profesionales y un ENUM anterior en documentos_profesional.

AGREGADO / CAMBIADO:
- Tabla profesionales ahora con:
  - tipo_documento_identidad ENUM('CI','NIT') NOT NULL
  - numero_documento (string)
  - resto de columnas (id_usuario, id_categoria, experiencia_anios, descripcion_servicio, zona_trabajo, estados...)
- Tabla documentos_profesional ahora con:
  - tipo_documento_archivo ENUM('CI_ANVERSO','CI_REVERSO','NIT','CERTIFICADO_TECNICO','REFERENCIA_LABORAL','OTRO') NOT NULL
  - archivo NOT NULL
  - estado_revision, observacion, fecha_subida
- Semilla (seed) de roles y categorías con INSERTs (cuidado con UNIQUE en re-ejecución).

====================================================
2) app/models/Profesional.php
====================================================
Se MODIFICÓ.

QUITADO:
- Método ciNitExiste($ciNit) (consulta por ci_nit).
- Inserción de profesionales usando:
  - tipo_documento
  - ci_nit

AGREGADO / CAMBIADO:
- Nueva validación/unicidad por numero:
  - numeroDocumentoExiste($numeroDocumento)
- crearProfesional($datos) inserta ahora en:
  - tipo_documento_identidad
  - numero_documento

====================================================
3) app/models/DocumentoProfesional.php
====================================================
Se MODIFICÓ.

QUITADO:
- Uso de documentos_profesional.tipo_documento al insertar.

AGREGADO / CAMBIADO:
- Uso de documentos_profesional.tipo_documento_archivo al insertar.

====================================================
4) app/controllers/AuthController.php
====================================================
Se MODIFICÓ.

QUITADO / dejado sin uso (para el profesional):
- Lectura/uso de:
  - $_POST['tipo_documento']
  - $_POST['ci_nit']
- Inserción del documento con tipo fijo 'CERTIFICADO_TECNICO' sin validar contra ENUM nuevo.

AGREGADO / CAMBIADO:
- Lectura de campos nuevos desde el POST:
  - tipo_documento_identidad (CI/NIT)
  - numero_documento
  - tipo_documento_archivo (para categorizar el archivo)
- Validaciones:
  - tipo_documento_identidad debe ser CI o NIT
  - rangos de longitud por tipo (CI: 5-12, NIT: 7-15)
  - unicidad de numero_documento
- Inserción del documento con fallback seguro:
  - si tipo_documento_archivo viene vacío o inválido => usar CERTIFICADO_TECNICO

====================================================
5) app/views/auth/registro_profesional.php
====================================================
Se MODIFICÓ.

QUITADO (en el formulario):
- Campo select:
  - name="tipo_documento"
- Campo input:
  - name="ci_nit"
- Select ampliado CI/NIT que no matcheaba con el ENUM real nuevo.

AGREGADO / CAMBIADO (en el formulario):
- Campos nuevos para identidad:
  - name="tipo_documento_identidad" (CI/NIT)
  - name="numero_documento"
- Campo nuevo para el archivo:
  - name="tipo_documento_archivo" (CERTIFICADO_TECNICO, REFERENCIA_LABORAL, CI_ANVERSO, CI_REVERSO, NIT, OTRO)

Nota:
- También se ajustaron variables PHP del view para evitar warnings por variables inexistentes.
